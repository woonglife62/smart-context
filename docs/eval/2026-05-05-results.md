## Triage Decisions (Phase E)

All 5 issues from the eval are classified **report-only** (§7.2 of the spec). None met the §7.1 criteria for an immediate fix (≤5 lines, ≤1 file, all tests still pass).

| Issue | Decision | Reason |
|---|---|---|
| P1: exact_identifier 0% top-1 | report-only | Use→definition inversion. `scripts/eval.js` outranks `src/searchEngine.js` because eval.js calls `smartContext()` many times. The ranker scores raw match density, so callers beat definers. Real fix is algorithmic (boost definition signals, demote import-only matches). |
| P1: conceptual 0% top-1 | report-only | Plan and spec docs (`docs/superpowers/...`) outrank `src/` files for queries about "tokenization" and "ranking". The 0.2x prose multiplier reduces but does not eliminate the problem because doc files contain many matches. Real fix is algorithmic. |
| P1: prose_only 0% top-1 | report-only | All 3 prose queries return rank 0 — the expected README/spec is buried under other markdown files. The 0.2x multiplier applies uniformly across prose, so larger plan docs beat smaller ones. Need intra-prose preference (path proximity, README weighting). |
| P1: distractor 0.67 < 0.7 | report-only | Only `distract-2` fails. The query "test fixtures" pulls plan docs first because the plan literally says "test/fixtures/golden-queries.json" many times. This is a self-eval circularity, not a tunable parameter. |
| P2: warm p50 (rg) 113 ms > 100 ms | report-only | Ripgrep startup overhead dominates on this small repo (~30 files). The Node fallback is faster (26 ms) here. Acceptable on larger codebases; profile in a future cycle. |

### Why no immediate fixes were applied

The 4 P1 ranking issues all share a single root cause: the score formula treats every match equally and has no notion of "definition vs. usage" or "primary doc vs. secondary doc". Fixing any one of them requires changing the scoring function in `src/ranker.js` by more than the §7.1 budget. Tightening the prose multiplier further would not help (the offenders for prose-only are themselves prose). Adding a `src/` boost would help conceptual but hurt distractor (`mcp/smart-context-server.js` is correctly the answer for one query). These tradeoffs need a redesigned formula, not a single-line tweak.

### Self-eval circularity (note)

Several false positives stem from the eval fixture and plan/spec docs containing the literal query text. For example, `edge-2` ("aaaaa") returns results because the string `"aaaaa"` is checked into `test/fixtures/golden-queries.json`, `docs/eval/2026-05-05-results.json`, and the plan/spec markdown. This is not a smart-context defect — it is inherent to using the repo's own files as the eval corpus. A future cycle could exclude `test/fixtures/golden-queries.json` and `docs/eval/` from the scanner default excludes, but that is a config change with broader implications.
